为了实现一个既能保护平台利益、又能“尊重用户”的高性能大模型网关，我们需要一套结合**实时状态追踪**、**两阶段计费**以及**补偿机制**的完整方案。

以下是针对 3 种模式与 3 种场景交叉下的完整计费处理逻辑：

---

## 核心计费状态机与矩阵

我们将计费流程分为：**预扣/锁定 (Lock) -> 执行 (Execute) -> 结算/回滚 (Settle/Rollback)**。

### 场景处理矩阵

| 请求模式 | 正常完成 (Normal) | 用户主动停止 (Active Abort) | 用户被动断开 (Passive Disconnect) |
| :--- | :--- | :--- | :--- |
| **1. 同步流式 (Sync Stream)** | 按厂商返回的 `usage` 或网关拦截到的完整 Token 计费。 | **按已发送给用户的 Token 计费。** 立即切断与厂商连接，释放余量。 | **暂不计费或按已发送计费。** 进入“待核销”状态，若用户未在短时间内重连/找回，则冲抵。 |
| **2. 同步非流式 (Sync Non-Stream)** | 成功交付完整响应后计费。 | **按厂商已生成的量计费。**（视为用户主动放弃已消耗资源） | **进入“缓存找回”流程。** 响应存入 Redis，过期未取回则回滚扣费。 |
| **3. 异步 (Async/Batch)** | 任务完成后，通过 Webhook 或轮询告知用户并计费。 | **按已消耗资源计费。**（如视频已生成 50%，按 50% 或全额计费，取决于厂商策略） | **无影响。** 异步任务继续执行，完成后在后台更新账单，用户下次登录可见。 |

---

## 详细技术实现方案

### 1. 预处理阶段：额度锁定 (Pre-locking)
无论何种请求，网关接入后的第一步是**预扣/锁定**。
* 根据请求参数（如 `max_tokens` 或视频时长）计算最高可能消费，锁定用户账户额度。
* 防止用户通过并发请求恶意刷爆账户。

### 2. 执行阶段：连接监控与错误分类
这是区分“主动”与“被动”的关键，需要在网关层捕获不同的信号：

* **信号捕获 (以 Go/Rust 为例)：**
    * **`ctx.Done()` / `Future Drop`**: 通常代表用户主动取消（如前端调用 `abort()`）或网关超时。
    * **`EPIPE` / `Broken Pipe` / `Connection Reset`**: 代表物理连接异常断开（被动）。
* **动作：**
    * 如果是**主动取消**：网关记录 `STATUS_ABORTED`。
    * 如果是**被动断开**：网关记录 `STATUS_DISCONNECTED`。

### 3. 同步非流式的“尊重”方案：缓存找回机制
对于同步非流式，由于是“全无或全有”，连接断开时最难处理。
1.  **静默接收**：当用户被动断开时，网关**不停止**接收厂商响应，而是完成后存入 Redis (TTL=5min)。
2.  **状态挂起**：该笔订单状态设为 `PENDING_DELIVERY`。
3.  **找回接口**：客户端检测到断连后，自动尝试请求 `/v1/recover/{request_id}`。
    * **找回成功**：从 Redis 返回数据，正式计费。
    * **找回失败 (超时)**：Redis 数据过期，网关发起回滚操作，**不计费**（因为用户确实没拿到结果）。

### 4. 同步流式的“公平”方案：实时拦截计数
1.  **双向流监控**：网关在转发每个 Chunk 时，内置一个计数器。
2.  **断开点记录**：
    * **主动停止**：记录断开瞬间已发送的 Token 数。由于是用户主观行为，按此数值扣费。
    * **被动断开**：网关将已生成的 Response 暂存。由于流式数据具有即时性，若断开发生在响应末尾，可视为成功；若发生在开头，则按“非流式”的找回机制处理，或直接退费。

### 5. 异步请求：影子核销 (Shadow Reconciliation)
异步请求（视频、批量）本身不依赖当前连接：
1.  **解耦连接**：请求提交成功即返回 `job_id`。
2.  **后台履约**：网关后台进程负责监控厂商状态。
3.  **离线计费**：无论用户是否在线，厂商返回 `usage` 后，网关自动扣除锁定额度的多余部分，完成结算。

## 异常兜底机制

* **超时回滚**：如果网关与厂商之间超时（非用户原因），锁定额度必须 **100% 自动原路退回**。
* **幂等 ID 重试**：所有计费操作携带 `request_id` 的幂等标识，防止重连找回时发生重复扣费。
* **日志存证**：网关详细记录断开时的错误栈（是 `EOF` 还是 `Reset`），作为后续用户申诉的证据。
